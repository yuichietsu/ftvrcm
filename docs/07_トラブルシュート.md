# トラブルシュート

## 1. アクセシビリティ設定にサービスが出ない（Fire OS）

Fire OSのビルドや端末設定によっては、設定UIから「ダウンロードしたサービス（Accessibility Service）」のトグルが見つからないことがあります。
ADBが使えるならシェルから有効化できることがあります。

1. Fire TVで`開発者オプション` → `ADBデバッグ`をON
2. PCから接続（同一LAN）
   - `adb connect <FIRE_TV_IP>:5555`
3. サービス有効化（コンポーネント名）
   - `adb shell settings put secure enabled_accessibility_services com.ftvrcm/com.ftvrcm.service.RemoteControlAccessibilityService`
   - `adb shell settings put secure accessibility_enabled 1`
4. 反映しない場合は再起動
   - `adb reboot`

有効化後、設定画面の「現在のモード」が切り替わることを確認してください。

注意：Fire TV（Fire OS）では BACK 長押しがシステム側で処理され（`com.amazon.tv.action.BACK_LONGPRESSED` 等）、アクセシビリティ自体がOFFに戻る場合があります。
その場合は、アプリの「モード切り替えキー」を`MENU`など別のキーに変更することを推奨します。

切り替えがまったく効かない場合は、アクセシビリティサービス自体がOFFになっている可能性が高いです。設定画面の「必要状態：アクセシビリティサービス」がOFFなら、上記のADB投入でONに戻してください。

## 2. ポインタは出るがタップが発生しない

### 2.1 まず確認すること

- タッチ操作モードに入ると、輪っかのポインタ（overlay）が表示されます。
- タップは、`AccessibilityService`からの注入（ジェスチャまたはノードクリック）で行います。

### 2.2 うまくいかない場合

- Fire OS側で、特定アプリやホーム画面（システムUI）への注入が制限されている場合があります。
- その場合、アプリ内/システムUIではクリックできても、他アプリ中は制限されるケースがあります。

補足：縦スワイプ（CH+/CH-）は、ジェスチャ注入が失敗した場合にスクロール操作（`ACTION_SCROLL_*`）を試行します。

## 3. 設定画面を開くとすぐ落ちる

`SettingsActivity`起動直後にクラッシュする場合、`logcat`に例外が出ていることがあります。

原因：`ListPreference`に`useSimpleSummaryProvider`を設定している状態で、コード側から`preference.summary = ...`のように`summary`を上書きすると例外になります。

対策：

- `useSimpleSummaryProvider`を使う場合はコード側で`summary`を上書きしない
- もしくは、コード側で管理したい場合は`useSimpleSummaryProvider`を外す

### 3.2 Fire TVで起動直後に落ちる（`RecyclerView`レイアウト中の`summary`更新）

症状：Fire TVで設定画面を開くと即終了し、`logcat`に次のような例外が出ます。

- `IllegalStateException: Cannot call this method while RecyclerView is computing a layout or scrolling`

原因：`Preference`の`onBindViewHolder`内で`summary`を更新し、`RecyclerView`のレイアウト中に`notifyItemChanged`が発生するためです。

対策：

- 最新のAPKをインストールしてください（`onBindViewHolder`では`summary`を直接更新せず、`summary`ビューへの反映に切り替え済み）。

## 4. プロキシ方式でカーソルだけ動いて操作できない

症状：`emulation_method=PROXY`にすると、カーソル移動はできるがタップ/スワイプが発生しない。

補足：Fire OS 8では「アプリから端末内`adbd`へのローカル接続」がブロックされる場合があります。そのため、従来の“ADB直結”方式ではなく、PC上のプロキシ経由で操作します。

### 4.1 接続先（proxy_host / proxy_port / proxy_token）を確認

プロキシ方式では、アプリはPC上のプロキシサーバ（HTTP）へ接続します。

- 設定 → `プロキシホスト`（`proxy_host`）にPCのIPを指定（例: `192.168.11.127`）
- `プロキシポート`（`proxy_port`）はプロキシ起動時の`PORT`（デフォルト`8787`）
- `プロキシトークン`（`proxy_token`）は、PC側の`PROXY_TOKEN`と一致させる（設定している場合）

### 4.2 PC側プロキシ/ADBの状態を確認

- PC側で`adb connect <FIRE_TV_IP>:5555`ができること
- PC側プロキシが起動していること（`/health`が`ok: true`で返ること）

補足：`proxy-server` は、ADBが未接続/オフラインの場合に `adb connect <serial>` を自動で試行します（`serial` が `ip:port` の場合）。
また、`unauthorized`（認証未承認）等が短時間に連続した場合は、負荷を避けるための制限（クールダウン/時間当たり上限）をかけた上で、ベストエフォートの再認証フロー（`disconnect`→`kill-server`→`start-server`→`connect`）を実行します。
ただし、端末側での認証許可（デバッグ許可ダイアログの承認等）はユーザー操作が必要です。

補足：アプリ設定の「プロキシ疎通テスト」は、現在の設定で`GET /health`を呼び出します。`/health`はADBの状態も確認するため、ADBが停止/未接続の場合は`non-2xx`で失敗します。失敗した場合は、その場でエラー詳細（HTTPステータス/例外メッセージ）を表示します。

また、タッチ操作モードへ切り替える際にADBプロキシが疎通できない場合は、エラーを表示して切り替えをキャンセルします（通常操作のままになります）。

補足：ADBプロキシ経由の操作が失敗した場合、操作フィードバックのアニメーションは表示されず、トーストでエラーが表示されます。

プロキシサーバはリポジトリ内の[proxy-server/README.md](../proxy-server/README.md)を参照してください。

### 4.3 `Cleartext HTTP traffic ... not permitted` が出る

症状：アプリ側の`last_gesture_detail`に次のようなエラーが記録され、PC側プロキシにリクエストが届きません。

- `IOException: Cleartext HTTP traffic to <IP> not permitted`

原因：Android（Fire OS含む）は既定でHTTP（暗号化なし）通信を禁止することがあります。

対策：本プロジェクトのプロキシは現状HTTPのため、アプリ側でクリアテキスト通信を許可する必要があります。

- 最新のAPKをインストールして再試行（`android:usesCleartextTraffic="true"`を含むビルド）
- もしくはプロキシをHTTPS化（将来的対応）

### 4.4 `401 unauthorized` になる

原因：PC側`PROXY_TOKEN`とアプリ側`proxy_token`が一致していません。

対策：

- PC側をアプリ側に合わせる（例: `export PROXY_TOKEN="<proxy_token>"`）
- もしくはアプリ側の`proxy_token`をPC側に合わせる

### 4.5 `NetworkOnMainThreadException` が出る

症状：FireTV側の`logcat`に次のようなログが出て、PC側プロキシに`POST /tap`や`POST /swipe`が届きません。

- `proxy /tap failed (NetworkOnMainThreadException)`

原因：Android/Fire OSではメインスレッドでネットワーク通信を行うと例外で失敗します。

対策：最新のAPKをインストールしてください（PROXY送信をバックグラウンドスレッドで実行する修正を含みます）。
